One orchestrator, two entry points

A Claude Code workflow that drives a task through its whole lifecycle with a fleet of specialized subagents. The orchestrator judges results, routes work, and talks to the human — every substantive step is delegated, and every human touchpoint is a fixed-format gate. Start cold with /new-task; follow up warm with /iterate.

Pick your lane

Two ways in

Whether a task is cold or a follow-up on already-reviewed work decides which command you run. Both share the same subagents, routes, gates, and review machinery — they differ in how much of the lifecycle they pay for.

/new-task The full lifecycle

For a cold task. Brainstorm → approach review → plan → plan review → implement → implementation review → CI → retrospective. Eight phases, four human gates, risk-scaled review, and a self-improvement pass that writes lessons back. On finish it leaves a run manifest so a later follow-up can start warm.

Phases 0–7 + 6.5 4 gates brainstorm → retro
Dive into /new-task → /iterate The warm-start follow-up

For a delta on a baseline /new-task already reviewed. Loads the run manifest, reuses the worktree warm, and runs a delta implement + delta review — no brainstorm, no plan-review, no from-scratch fan-out. One gate, and the retrospective is deferred and batched across the whole session.

How they connect

The manifest is the handoff

Every finished /new-task writes a run manifest — route, SHAs, artifact paths, outcome, and open Should-fix findings. That file is exactly what /iterate loads to start warm, so follow-up work never re-discovers context the first run already established.

/new-task → finishes, writes run manifest → /iterate → warm delta ↺ (loops, batched) → one retro at session end
