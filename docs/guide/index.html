One orchestrator, many lanes

A Claude Code workflow that drives work through its whole lifecycle with a fleet of specialized subagents. The orchestrator judges results, routes work, and talks to the human — every substantive step is delegated, and every human touchpoint is a fixed-format gate. Author with /new-task + /iterate, turn an issue into a plan with /triage-issue, understand code cheaply with /explain, review a teammate's PR with /review-pr, and act on inbound review comments with /address-review — all on the same fleet.

Pick your lane

Two ways in

Whether a task is cold or a follow-up on already-reviewed work decides which command you run. Both share the same subagents, routes, gates, and review machinery — they differ in how much of the lifecycle they pay for.

/new-task The full lifecycle

For a cold task. Brainstorm → approach review → plan → plan review → implement → implementation review → CI → retrospective. Eight phases, four human gates, risk-scaled review, and a self-improvement pass that writes lessons back. On finish it leaves a run manifest so a later follow-up can start warm.

Phases 0–7 + 6.5 4 gates brainstorm → retro
Dive into /new-task → /iterate The warm-start follow-up

For a delta on a baseline /new-task already reviewed. Loads the run manifest, reuses the worktree warm, and runs a delta implement + delta review — no brainstorm, no plan-review, no from-scratch fan-out. One gate, and the retrospective is deferred and batched across the whole session.

How they connect

The manifest is the handoff

Every finished /new-task writes a run manifest — route, SHAs, artifact paths, outcome, and open Should-fix findings. That file is exactly what /iterate loads to start warm, so follow-up work never re-discovers context the first run already established.

/new-task → finishes, writes run manifest → /iterate → warm delta ↺ (loops, batched) → one retro at session end

Before the lifecycle

Turning an issue into a plan

The entry points above assume you already know what to build. /triage-issue is the step before — it classifies a raw GitHub issue, reproduces the bug, root-causes it, and scopes the fix, then stops before any code. It writes a triage manifest that warm-starts /new-task, so issue → triage → /new-task → /iterate is one pipeline.

/triage-issue The front door

For a raw issue, before you know what to build. Classifies (bug / feature / question / needs-info / duplicate), routes it, runs the Phase 0/1 investigation, proves a repro per verify-fix, and root-causes with file:line evidence — read-only, no fix. Delivers a scoped plan + a ready-to-paste /new-task invocation and a triage manifest.

Just understand it

The cheap read-only lane

Not every question needs the lifecycle. /explain answers "how does X work / where is Y / why is Z this way" for the price of a few haiku searches — it classifies the question into one of seven cases and returns a grounded, cited answer in that case's fixed shape. No worktree, no gates, no writes.

/explain The cheap read-only lane

For understanding, not changing. Classifies into 7 cases (locate · mechanism · flow · architecture · impact · compare · why), sweeps with parallel searchers on haiku/low effort, and synthesizes inline — no architect/opus. Every claim carries a file:line; anything absent is "not found," never invented.

Phases E0–E2 7 cases haiku default

Reviewing inbound work

Reviewing work you didn't author

Both entry points above review work you just wrote. Two more lanes handle review traffic with other people: /review-pr reviews a teammate's pull request (read-only, outbound), and /address-review receives the comments human reviewers leave on a PR you authored — driving every thread to an explicit disposition, with the same skeptic filter applied to human suggestions before any coder spends tokens on them.

/review-pr The read-only review lane

For a PR you didn't author. Derives the PR's intent in place of a plan, reads CI status instead of running the branch, then runs the risk-scaled review fan-out, confidence-scored consolidation, and default-refute skeptic pass. Delivers a three-section report (Must-fix · Should-fix · Rejected) locally by default; --comment posts it behind one confirm gate.

Phases R0–R3 0 gates (local) read-only
Dive into /review-pr → /address-review The inbound review lane

For the comments on a PR you authored. Ingests every unresolved thread, skeptic-checks each defect claim before coder spend, fixes accepted asks through the /iterate delta lane, and gates a per-thread disposition table — fixed · answered · declined · clarify · handoff — so no thread is silently skipped and no push-back is auto-posted.

Phases A0–A3 1 gate (GATE A) 6 dispositions

Around the lifecycle

Maintaining and measuring the workflow

The four commands above do the work. Two more keep the workflow itself healthy — one runs on a schedule to sync self-improvements and curate the learnings memory, the other scores the workflow so complexity is added or cut on evidence, not intuition. Both operate on the workflow files, not on your task, so they have no lifecycle pages of their own — their contracts live in the command source.

/workflow-maintenance The scheduled caretaker

Idempotent and safe to schedule (local cron or a remote routine). Commits faithful capture.sh diffs of live agent/command/skill self-improvements (never learnings — those stay local), proposes — never applies — local learnings prunes, flags live bullets that duplicate a promoted skill, and reports red trunks. Run it periodically to keep repo and live copies in sync.

idempotent schedulable capture · curate · report
See commands/workflow-maintenance.md →
/workflow-eval The measure-and-iterate harness

Scores the workflow before you add or cut complexity. --lint-only runs the deterministic consistency check (run any time); a full run scores /new-task against frozen fixture tasks and writes a dated scorecard; --variant ablates a layer for an A/B verdict on whether it earns its cost; --contracts tests each agent against its declared contract. Expensive and opt-in — never auto-scheduled.

lint · score · ablate opt-in scorecard diff
