/iterate — the warm-start fast lane

When a change has already been carried through /new-task's review, tweaking or extending it shouldn't re-pay the full lifecycle. /iterate treats the follow-up as a delta on a reviewed baseline: no brainstorm, no plan-review, no from-scratch review fan-out, and no per-tweak retrospective.

The warm-start flow

Three phases, one gate

Left to right: load the baseline, implement the delta, review only what changed, decide. It reuses /new-task's review-loop conventions, escalation ladder, and route model verbatim — only the warm-start seam and the deferred retro are new.

Phase I0
Warm setup

Load the prior run's manifest — route, SHAs, design/plan/retro paths, outcome, and carried open Should-fix. Reuse the worktree warm if it still exists. Seed context from the manifest instead of re-running searcher / researcher. Inherit the route as a floor (monotonic — escalate only).

→
Phase I1
Delta implement

ONE architect writes a delta plan-lite against the seeded baseline; carried Should-fix fold in. coders implement in parallel where independent, delegating test runs to test-runner. Bug-fix deltas proven per verify-fix.

→
Phase I2 ↻ max 5
Delta review

The baseline is trusted, so review is delta by construction: one fresh reviewer over the iteration diff, seeded with the carried Should-fix checklist. An escalation to high-stakes triggers the full Phase 6 fan-out instead. Verdict computed: PASS ⇔ green ∧ zero Must-fix ∧ verified.

→
GATE I
One gate

Same four-section decidable summary, rendered as plain text that ends the turn. Auto-approves on fully-green criteria and no route escalation — marked auto-approved, never silent. On approval, finish the branch or push to the manifest's PR (then Phase 6.5 CI applies).

⇄Retro is deferred & batched

No per-iteration Phase 7 — that tax is the whole reason /iterate exists. Each iteration appends a compact entry to the manifest's log and bumps its HEAD_SHA. At session end (or --finish), Phase 7 runs once over the whole session — one retro, one curated GATE 4, keyed to the batch, not to each tweak.

⤴When to bounce to /new-task

No readable manifest, or the request isn't a delta on that baseline — a new subsystem, an unrelated area, a greenfield feature, or a delta that is itself ≥200 lines of new surface → /iterate stops and hands you back to /new-task. It assumes an already-reviewed baseline; without one there's nothing to delta against.

The handoff

What the run manifest carries

Written by /new-task Phase 6 on approval, the manifest is the single artifact /iterate reads to start warm. It is additive — writing it changes no gate decision.

routeFinal route

The floor /iterate inherits — it may only escalate from here.

shaBASE / HEAD SHA

The reviewed baseline. /iterate's BASE_SHA = this HEAD_SHA.

treeWorktree + branch

Reused warm if it still exists; otherwise recreated from the branch / SHA.

pathsDesign · plan · retro

The seeded context, so legwork isn't re-run for what the first pass covered.

outOutcome

merged · PR #N · local-kept — decides how the branch is finished.

todoOpen Should-fix

Findings the first run didn't fix, carried as the delta-review checklist.

Not reinvented

What it borrows from /new-task

/iterate adds only the warm-start seam and the deferred retro. Everything else is the same machinery — read it once on the /new-task page.
