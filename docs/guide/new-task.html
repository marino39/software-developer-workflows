/new-task — the full-lifecycle task workflow

The cold-start entry point. One orchestrator drives a task from brainstorm to retrospective — judging results, routing work, and talking to the human, while every substantive step is delegated to a subagent and every human touchpoint is a fixed-format gate. Already-reviewed work? Follow up on the warm-start /iterate lane instead.

The pipeline

How a task flows through the workflow

The spine runs top to bottom. Teal nodes are phases; circular amber nodes are human gates. Scoped tasks take the dashed fast path, collapsing phases 1–4. Review phases (2, 4, 6) are bounded loops that never silently pass.

Phase5

Implement

Each plan task goes to a coder; independent tasks run in parallel. Coders route test runs to test-runner so raw output never reaches the orchestrator. Stuck after two attempts → escalate to debugger.

Phase6

Implementation review ↻ max 5

Behavioral verification runs first. Then a risk-scaled fan-out: small non-high-stakes diffs get a reduced tier; everything else gets the full tier — superpowers review, codex, and three lens reviewers. A fresh consolidator scores and dedupes findings; each Must-fix survives a default-refute skeptic pass. Verdict is computed: PASS ⇔ tests green ∧ zero Must-fix ∧ verified. On approval it writes a run manifest — route, SHAs, paths, outcome, and open Should-fix — so a later /iterate can warm-start.

verify-feature requesting-code-review reviewer ×N lenses codex (full tier) consolidator + skeptics run manifest → /iterate
Gate3

GATE 3 · Implementation auto-approves on the fast path

Gate summary: final route, files changed, test + behavioral-verification status, fan-out tier, per-channel finding counts, Must-fix / Should-fix. On a non-escalated scoped route with fully green criteria, this gate auto-approves — still emitting the full summary, marked auto-approved. On approval → finish the branch (merge / PR).

The cast

Seven subagents, each with a fixed model tier

The orchestrator never does substantive work itself. Agents are tools with input/output contracts; each returns a capped structured summary, never a raw transcript. Model tier is the default — the escalation ladder overrides it one rung at a time.

AgentDefaultRoleEscalates to
searcher haiku Fast codebase lookups — definitions, callers, usages, config. Answers the exact question, ≤200 words. sonnet
researcher sonnet External research — docs, APIs, prior art. Returns synthesized findings with sources and confidence. opus
architect opus Designs plans — file-level steps, interfaces, trade-offs. Writes no code; may spawn only searcher / researcher. fable
coder sonnet Implements the plan exactly, step by step. Delegates test runs to test-runner; proves bug fixes before reporting. → debugger (no model bump)
reviewer sonnet Reviews a diff against the plan — correctness, edge cases, conventions. Read-only; PASS/FAIL + numbered issues. opus → fable
debugger opus Escalation-only. Root-causes stubborn failures — reproduce, isolate, verify hypothesis — then fixes minimally. fable
test-runner haiku Runs builds / tests / CI-log digests and reports compactly. Never fixes anything; raw output stops here. —

Human contract

Four gates, one fixed format

Work-shaping input is allowed only in phases 1, 3, and the fast-path scope gate. Every other touchpoint is a gate: a ≤25-line summary with four sections — Results · Key decisions · Deviations · Next — then a single approve / reject. A gate must be decidable: its Results carry the decision evidence inline, never just a link to open.

GATE 1
Approach
Chosen approach, rejected alternatives, remaining risks. After phase 2.
GATE 2
Plan
Plan steps and deviations vs the approved design. After phase 4.
GATE 3
Implementation · fast-path auto-approve
Route, files, tests, verification, fan-out tier, Must-fix counts. After phase 6.
GATE 4
Self-update
Proposed learnings / workflow edits shown as a diff. After phase 7.

▤Rendered as plain text, then the turn ends

Every gate — GATES 1–4, GATE I, and the scope gate — is presented as a single plain-text message whose body is the full summary, then the turn ends. It is never collected through AskUserQuestion, whose picker would hide the evidence behind the widget. The summary stays the last visible text.

?A clarifying question ≠ a decision

A clarifying question is allowed at any gate — it is how the human makes the gate decidable. It is answered in-band and the summary is re-presented; a question never counts as approve / reject and never advances or returns the phase. Only an explicit approve / reject moves the workflow.

Shared machinery

The parts that make it reliable

↻Bounded review loops

Phases 2, 4, and 6 share one delta-review engine. Iteration 1 runs the full review; iterations 2+ hand a fresh reviewer the prior findings as a checklist and scan only revised sections. Cap is 5 — exhaustion halts with a failure digest at the gate, never a silent pass.

review→ numbered issues?→ revise ↺→ exit on clean

⇧Escalation ladder & fable budget

Same error twice → escalate one model rung with fresh context. Different error each time → the upstream artifact is wrong; go back a phase instead.

  • Two fable slots are by-design & unbudgeted: the phase-1 synthesizer and the phase-2 adversarial reviewer.
  • Beyond those, at most one fable escalation per run — debugger, reviewer, or architect.
  • Ladder exhausted → STOP with a full failure digest.

⌖Risk-scaled review fan-out

Phase 6 sizes its review to the diff. Small, non-high-stakes changes get a reduced tier (one combined reviewer, codex skipped, orchestrator consolidates). Larger or high-stakes changes get the full tier.

  • Channel A — superpowers code review.
  • Channel B — codex CLI (full tier, degrades gracefully).
  • Channel C — three lens reviewers: bug scan · git history · CLAUDE.md compliance.

⚑Skeptic pass & computed verdict

A fresh consolidator dedupes findings, scores each 0–100 confidence, drops <50, and tags Should-fix (50–79) vs Must-fix (≥80). Each Must-fix then faces a default-refute skeptic — it survives only if the skeptic can't refute it.

PASS ⇔ tests green ∧ zero Must-fix ∧ behavioral verification passed

Memory & self-improvement

The workflow improves itself, under review

Phase 0 loads only the learnings whose trigger tags match the task; phase 7 writes new ones back. Every self-edit passes GATE 4 and is versioned in git — hand edits and run-time self-improvements share the same source of truth.

Token hygiene

Why the orchestrator stays cheap

∅No raw transcripts

Every subagent returns a capped structured summary. The orchestrator never ingests raw agent output.

∥Parallel fan-out

Independent subagents run in a single message; dependent ones run in order.

▷Tests via test-runner

All builds, test suites, and CI logs are digested by test-runner — raw logs never reach the orchestrator's context.
