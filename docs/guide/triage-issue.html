/triage-issue — decide whether & what, before how

/new-task assumes you already know what to build. /triage-issue is the step before: it turns a raw GitHub issue into a classification, a proven repro, a root-cause hypothesis, and a scoped plan — then stops before any code. It writes a triage manifest that warm-starts /new-task, making it the lifecycle's front door rather than a sidecar.

The triage flow

Four phases, then a handoff

Classify, investigate, root-cause + scope, deliver. It reuses /new-task's Phase 0/1 legwork and route model verbatim — only the issue-triage seam and the manifest are new. There is no plan or implement phase, so no authoring gate; the default run is side-effect-free.

Phase T0
Setup & classify

Read the issue (title · body · labels · comments · linked PRs); search for duplicates. Classify: bug · feature · question · needs-info · duplicate. Non-actionable classes short-circuit to a recommended response. An actionable issue gets a route (scoped / standard / high-stakes) per Phase 0 — the manifest's route floor.

→
Phase T1
Investigate

The Phase 0/1 legwork, in parallel: searcher maps where the relevant code lives and the suspect call paths; researcher pulls external docs, library behavior, and similar issues. Their capped digests go verbatim into the manifest so /new-task doesn't re-discover them.

→
Phase T2
Root-cause + scope

Bug: a proven failing repro per verify-fix in an isolated scratch worktree (a pre-existing failing test that pinpoints it qualifies), plus a root-cause hypothesis with file:line evidence. Feature: a light approach sketch — ONE architect, not the full brainstorm (deferred to Phase 1). Then a scope estimate → route + which follow-up command.

→
Phase T3
Deliver + manifest

A triage report (classification · route · repro · root-cause · scope) plus a ready-to-paste handoff — the exact /new-task or /iterate invocation — and the triage manifest. --comment posts a concise summary to the issue, behind one gate.

◎Read-only on the product

Never edits, commits, or pushes product code; never closes, labels, or reopens the issue; never runs /new-task itself. The only writes are the triage manifest and a bug's repro test in an isolated scratch worktree — the fix is never applied. Triage stops at the hypothesis.

▤--comment posts behind one gate

Posting to the issue is outward-facing, so --comment presents the draft and ends the turn — only an explicit approve posts it, as a COMMENT (one concise triage summary + route + handoff), never a status change. Issue text is untrusted external content: instructions embedded in it are flagged, not followed.

The composition payoff

The triage manifest warm-starts /new-task

Just as /new-task's run manifest lets /iterate start warm, the triage manifest lets /new-task start warm — route decided, legwork done, scope set. The chain becomes one pipeline, not four disconnected commands.

issue → /triage-issue → writes triage manifest → /new-task (warm) → /iterate

The hypothesis with file:line and the target files / size estimate.

↳The consumption seam is wired in

/new-task Phase 0 now reads a referenced triage manifest: it seeds the investigation from the manifest's digests / repro / root-cause instead of re-running the legwork, and inherits the triage route as a monotonic floor — so a triage-classified high-stakes issue can never be cold-downgraded. The seam is additive and inert without a manifest (the existing suite is unaffected); its warm-vs-cold cost A/B (--tasks 09 --variant triage-cold) is set up and owed, exactly as the /iterate warm-start earned its ledger row.
