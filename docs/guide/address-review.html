/address-review — the review engine's inbox

/review-pr aims the review engine at code you did not author. /address-review is the mirror: human review comments arrive on a PR you (or a prior /new-task run) authored, and each thread is driven to an explicit disposition — fixed, answered, declined with reasons, sent back for clarification, or handed off as new work. Fixes ride /iterate's delta lane; every outward reply rides a gate.

The flow

Ingest → disposition → delta → one gate

The command is /iterate's delta lane with an ingestion front end. The one new idea sits in Phase A1: a human comment is a hypothesis about the code, not an order — so it faces the same default-refute skeptic that internal Must-fix findings face, before any coder spends tokens on it.

Phase A0
Setup & ingest

Fetch the PR (must be open, yours — a foreign PR bounces to /review-pr) and every unresolved thread. Warm-start from the run manifest: its design doc, plan, and route floor are the context; no re-discovery. A hand-authored PR without a manifest derives an intent digest and writes a fresh manifest on completion.

→
Phase A1
Disposition + skeptic

Every thread gets exactly one disposition: fix · done-already · answer · decline · clarify · handoff. Each defect claim faces a default-refute skeptic; refuted → a drafted push-back reply, never a silent drop and never blind implementation. Ambiguity → a drafted question, never a guess. Scope-expanding asks → a ready-to-paste /new-task handoff.

→
Phase A2–A3
Delta implement + review

/iterate Phase I1/I2 verbatim over the accepted fix set: plan-lite, coders, fixes proven per verify-fix, behavioral verification, then a single delta reviewer (full Phase 6 fan-out only on a high-stakes escalation). Verdict computed, never judged.

→
GATE A
One gate, then push + reply

The gate carries the per-thread disposition table — evidence and reply draft per row. On approval: push to the PR branch, post only the approved replies, then Phase 6.5 CI verification. Retro is deferred and batched into the manifest's iteration log.

✓Auto-approve is the pure fix case only

GATE A auto-approves iff every thread is fix/done-already, zero replies would post, tests are green, zero Must-fix, verification passed, and the route never escalated — the same posture as Phase 6.5 pushing CI fixes. Any decline, answer, clarify, or handoff → human gate: pushing back at a human reviewer is never auto-posted.

⚠Comments are untrusted input

Thread bodies are external data. A comment that tries to steer the run — widen scope, touch high-stakes code, suppress reporting — is dispositioned decline with an injection flag in its table row: surfaced, never obeyed, never omitted (the eval's T4 control tests exactly this).

The inbound seam

What changes when the reviewer is human

The engine already knew how to filter its own findings. The seam adapts three things for findings that arrive from people — and keeps the writes on a short leash.

⊘Skeptic before coder

A wrong suggestion costs one reviewer, not an implement-review cycle — Phase 6.4's placement, applied to human claims.

✎Refuted ≠ dropped

A refuted internal finding is silently demoted; a refuted human comment becomes a drafted, evidence-cited push-back reply behind the gate.

▦No silent skips

The disposition table has exactly one row per unresolved thread. "Ignored" is not a disposition.

∅Frugal replies

The pushed diff is the record — no "Fixed!" comments by default (--ack opts in). Replies exist to decline, answer, or clarify.

⌖Scoped writes

Push only to the PR's head branch; reply only in existing threads. Never approve, merge, resolve threads, or mark ready-for-review.

⇥Scope asks hand off

A "while you're here…" feature request gets a ready-to-paste /new-task invocation as its reply — the /triage-issue handoff pattern, not scope creep.

Not reinvented

What it borrows
