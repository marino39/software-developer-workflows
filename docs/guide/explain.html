/explain — classify, sweep cheap, ground every claim

The suite's lightest lane: no worktree, no lifecycle, no gates, no writes. /explain answers "how does X work / where is Y / how is Z wired / why is W this way" by classifying the question into one of seven cases and returning a grounded answer in that case's fixed output shape. Built to be cheap: the workhorse is searcher on haiku / low effort, fanned out in parallel, and the orchestrator synthesizes inline — no synthesis subagent, no architect/opus.

The flow

Classify → sweep → synthesize inline

Three light steps. The fan-out width comes from the case, not from crawling; synthesis happens in the orchestrator, which only ever ingests capped digests. That's the whole cost story.

Phase E0
Classify & scope

Resolve the question to exactly one primary case (table below) from its phrasing + intent; a mixed question folds a secondary in, and a pure external-library question routes to researcher or is bounced — never guessed. Pick the case's angles and fan-out width (shallow by default; --deep widens).

→
Phase E1
Sweep (parallel, cheap)

Launch the case's angles as parallel searchers in one message — haiku / low effort, each returning capped findings with file:line. A multi-modal sweep: each angle is blind to the others, so breadth is parallelism, not depth. Escalate to sonnet only on empty/off-target twice; add researcher only for external docs.

→
Phase E2
Synthesize inline

The orchestrator writes the answer directly from the digests — no synthesis subagent (the main cost lever) — in the resolved case's fixed output contract, prefixed by a scope note + classification header. --save also writes it to a path.

$Cheap by construction

searcher (haiku/low) is the only default tier — no architect/opus, an explanation doesn't need the deep-reasoning tier. Synthesis is inline, so there's no extra subagent round-trip. Shallow by default; --deep is the one knob that widens the sweep.

✓Grounding contract

Applies to every case: each claim carries a file:line (or a commit sha for rationale); anything absent is reported as "not found in the searched scope," never invented. A scope note opens each answer so gaps are visible — the anti-hallucination guarantee.

The structure

Seven cases, each a fixed output shape

Classification isn't decoration — it binds the answer to a contract, so the same kind of question always yields the same kind of output. Diagrams are spent only on the two structural cases.

#CaseClassified fromOutput contract (fixed shape)
1Locate"where is / what defines / where's X configured"Ranked location list — each path:line + one-line role. No prose.
2Mechanism"how does X work"Walkthrough — summary → numbered steps (each file:line) → inputs/outputs → edge cases & invariants.
3Flow / trace"what happens when / trace A→B / lifecycle of"Ordered sequence — entry → hops (each file:line) → exit; + Mermaid sequence diagram.
4Architecture"structure of / map / overview of W"Component map — table (component → responsibility → path) → connections → entry points; + Mermaid diagram.
5Impact / usage"what calls X / who depends on / blast radius of Y"Dependents list — direct callers (file:line) → transitive → blast-radius summary.
6Compare"difference between X and Y / A vs B"Comparison table — dimension → X → Y (cited) → when-to-use.
7Why / rationale"why is X this way / why not Y"Rationale list — each point sourced (comment file:line, doc, or commit sha); explicit "no recorded rationale found" when the code is silent.

?The Why case is the one Explore can't do

Cases 1–6 are faster, structured versions of what a raw search agent gives you. Case 7 is different: it reads CLAUDE.md + code comments + git history (log/blame) to explain why the code is the way it is — and says "no recorded rationale found" rather than rationalize. That grounded "why," refusing to invent intent, is the case ad-hoc exploration doesn't give you.
