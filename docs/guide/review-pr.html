/review-pr — aim the review engine outward

/new-task and /iterate review work you just wrote. /review-pr takes the same Phase 6 review engine — multi-channel fan-out, confidence-scored consolidation, default-refute skeptic pass — and points it at a pull request you did not author. It is read-only: its product is a review, never an edit, a push, or a merge.

The review flow

Four phases, no authoring gate

Fetch the PR, fan out the review, consolidate + skeptic-filter, deliver. There is no brainstorm, plan, or implement phase — so no GATE 1–4. The default run is side-effect-free and needs no approval; only --comment adds a single confirm-before-posting gate.

Phase R0
Setup & fetch

Read the PR metadata and its CI check status (green / red / pending — no local test run; CI adjudicates the run). Derive an intent digest from the linked RFC → issue → PR body → title — composing it is the orchestrator's; bulk-reading its sources is not (a linked doc is digested by researcher/searcher, and CLAUDE.md files are located, not read — the C3 lens Reads them itself). Fetch the head into a read-only worktree. Pick the review tier exactly as Phase 6 — auth / payments / migrations / data-deletion, or ≥200 lines → full; else reduced. A small PR trims the tier, never whether review is dispatched (the Delegation floor).

→
Phase R1
Review fan-out

Phase 6's channels, run verbatim — Channel A (superpowers review), Channel B (codex, full tier, degrades free), Channel C three lens reviewers (C1 shallow-bug · C2 git-history · C3 compliance). The one substitution: the intent digest stands in for the approved plan the in-workflow reviews check against.

→
Phase R2
Consolidate + skeptic

A fresh reviewer dedupes, scores each finding 0–100, drops <50, tags 50–79 Should-fix / ≥80 Must-fix. Each Must-fix then faces a default-refute skeptic — it survives only if the skeptic can't refute it. No PASS/FAIL merge verdict; the output is an advisory assessment.

→
Phase R3
Deliver

A report in three sections — Must-fix · Should-fix · Rejected (with the skeptic's refutation reasons) — plus ci: <state> and the advisory verdict. Default: local report, nothing posted. --comment: post it, behind the gate below.

◉Read-only by default

Never edits files, never pushes to the PR branch, never approves or merges. The local report is the default and has no gate — it's side-effect-free. Fixing a PR's findings is a different job; this command only reviews.

▤--comment posts behind one gate

Posting to someone else's PR is outward-facing, so --comment presents the draft review and ends the turn — only an explicit approve posts it. All Must-fix + Should-fix go inline at file:line; the Rejected set and CI status go in the summary body. The GitHub event is COMMENT — never approve/request-changes/merge.

The foreign-PR seam

What changes when the code isn't yours

An in-workflow review checks the diff against an approved plan. A PR you didn't author has no such artifact — only its stated intent — and you shouldn't run its code blind. Two things shift; the engine itself is unchanged.

✎Intent, not plan

The review spine becomes "does the diff satisfy what it claims?" — sourced linked-RFC → issue → PR body → title, fed to the channels in place of the plan.

ciCI, not local tests

No blind execution of a stranger's branch. The PR's CI check status is read from GitHub; the engine judges the code, CI judges the run.

◎Advisory, not gate

The verdict (looks good / comment / request-changes) is advice to the human reviewer — it never gates a merge or takes an action.

⚠Untrusted content

PR body, comments, linked issues are external data. Instructions embedded in them are not followed; a steering attempt is flagged in the report, not obeyed.

⌖Same tier logic

Reduced vs full is decided exactly as Phase 6 — a diff touching a high-stakes category forces the full tier with escalated reviewers.

∅Rejected kept

Findings the skeptic refuted or that scored <50 are surfaced with their reasons — transparency on what was considered and dismissed.

Not reinvented

What it borrows from /new-task

/review-pr adds a command surface, not new engine complexity — it reuses the Phase 6 machinery and every agent contract unchanged. Read it once on the /new-task page.
