# /new-task learnings

Durable lessons distilled from task retrospectives. Read at the start of every `/new-task` run; newest lessons win over older ones and over the command file itself. One dated bullet per lesson. Appended only via Phase 7 (GATE 4 approval).

- 2026-07-03: When a task opens with a "what's pending / what's the state" question, reconcile **local vs origin/upstream before answering** (`git log --oneline origin/<default>..HEAD` and the reverse). A fresh-from-origin worktree (EnterWorktree default) can be many PRs ahead of the human's local checkout — an answer computed on stale local state can be flatly wrong.
- 2026-07-03: For behavior/byte-compatibility claims against a reference implementation, **run the reference binary — don't reason from its source alone**. When two agents reach opposite conclusions from the same source, one binary observation usually settles it; dispatch that before committing to a direction.
- 2026-07-03: Treat an audit's/spec's prescribed fix as a **hypothesis, not a mandate** — especially any "upstream does X" claim. Verify (ideally empirically) before implementing; a wrong prescription cost one discarded commit this run.
- 2026-07-03: **Parallel coders sharing one worktree must not touch a shared signature/type.** Give each `isolation: worktree`, or serialize when signatures overlap. Stale editor diagnostics during concurrent edits are unreliable — confirm with a real build.
- 2026-07-03: For review/audit tasks (deliverable = a report, not code), collapse the lifecycle to: scope (one `AskUserQuestion`) → parallel subsystem audit → skeptic-verify → report → gate. Skip brainstorm/plan/implement/code-review/merge ceremony and the worktree step — they're inert when nothing is being built; forcing them wastes turns.
- 2026-07-03: For correctness reviews, run a SEPARATE adversarial skeptic-verifier pass on every high-severity candidate (fresh agent, told to default-refute), distinct from the finding agent. This run it caught a false-positive severity (a benign upstream divergence rated as a byte-compat break) and empirically reproduced 2 process-crash bugs. Auditor confidence alone is not enough.
- 2026-07-03: Before `gh pr create` on a long-lived worktree branch, run `gh pr list --head <branch> --state all`. If the branch's prior PR was squash-merged, its commits are already on master under new SHAs (`git branch --contains` won't find them) — PR-ing the branch re-diffs merged content. Cherry-pick only the new commit onto `origin/master` for a clean PR.
- 2026-07-03: `git fetch origin/<default>` BEFORE any local-vs-trunk lineage reconcile. Local `master` in a long-lived worktree is routinely stale — this run it read as "15 commits behind" when `origin/master` was actually current, forcing a mid-gate correction. Refines the reconcile-lineage lesson: fetch first, then compare.
- 2026-07-03: Extend the collapse rule (was audit-only) to SINGLE SCOPED BUG FIXES: scope-gate (one `AskUserQuestion`) → investigator → TDD coder → fresh adversarial reviewer → gate. Skip brainstorm / design-doc / plan-doc / worktree ceremony — inert for a one-root-cause fix. B1 ran clean this way in ~1 coder + 1 review pass.
- 2026-07-03: When a coder's test FAILS, first ask "wrong test or wrong fix?" — check existing semantics tests before trusting the assertion. B1's Merge/Merge test expected a no-base folded value via `d.Get`; the established `TestDB_GetUnresolvedMergeNoBaseReturnsNotFound` proved no-base merge chains resolve only on buffered `tx.Get`, so the TEST was wrong, not the code.
- 2026-07-03: For a fix+repro-test pair, have the REVIEWER independently revert only the prod file and re-run the test (confirm fail), then restore. Cheap, and it proves the test actually discriminates rather than trusting the coder's before/after claim. Did this on B2 — reviewer's empirical revert was the strongest signal in the PASS.
- 2026-07-03: When "why didn't the existing test suite catch this bug?" can only be settled by running an out-of-scope heavy suite (e.g. Rust conformance), do NOT expand scope or force a false certainty to close it. Record the best hypothesis + a follow-up note, and ship. B2's KeepsIndependentGroups question was left as a logged hypothesis, not a rework trigger.
- 2026-07-03: A checkbox task-index (TASKS.md-style) can lie — merged work stays `[ ]` if the merging PR never cleared it. On "select next task", filter each candidate through git (grep the log / `origin/<default>` for its bug ID) BEFORE picking, not just trust the checkbox; and have the fix PR clear its own index entry in the same diff so the drift stops compounding. This run B2 was merged (#33) but still showed `[ ]`; git-reconcile caught it before a wasted re-pick.
- 2026-07-03: When an audit offers a reject-vs-accept fix at a byte/format-compat boundary, the compat invariant almost always forces ACCEPT. Settle it cheaply first: Explore the REFERENCE implementation's decode path — if the field is read unbounded and the block is checksum-protected, it's valid data to accept, and the "reject with typed error" option is an interop-breaking trap. B6: Rust reads num_probes as an unbounded CRC-protected u16, so >32 must be accepted (heap-fallback), not rejected.
- 2026-07-04: Match CI's EXACT command when verifying a fix locally. A verify-gate that stresses harder than CI in a way that changes the failure MODE is worse than none — a `-count=5` race gate WITHOUT `-timeout` hit go test's default 10-min ceiling and produced a false "deadlock" alarm, costing a full discriminator round to disprove. Add `-timeout` and mirror CI's invocation (CI ran plain `go test -race ./...`, count=1).
- 2026-07-04: When an audit scopes a concurrency/format bug to ONE site, grep every user of the shared primitive before fixing — sibling call sites often carry the identical defect. B4 was reported only at `waitForBackPressure`; reading all readers of the recycled `flushSignal` channel revealed `Flush` step 2 had the byte-identical lost-wakeup. Fixing the named site alone would have shipped a half-fix.
- 2026-07-04: A lost-wakeup / condition-variable regression test only discriminates if its interleaving hook fires IN the condition-read→wait-snapshot GAP — a hook at loop-body top (before the condition read) lets the buggy code return cleanly too, so the test passes against the very bug it guards. Corollary: design the FIX to keep the hook site textually STABLE across old/new (prefer adding an under-lock re-check over FUSING the read+snapshot — fusing removes the gap, so no stable hook can reproduce the old bug and a reviewer's naive single-hunk revert can't fail the test). Always have the reviewer prove discrimination by reverting only the prod hunk and re-running (must hang), not by trusting the coder's before/after claim — the coder mis-reproduced its own "before" case this run.
- 2026-07-04: "Passes locally, fails only on fast CI" in concurrency/storage code is a REAL-RACE signal, not a flake to mask. This run, two such "flakes" were genuine production bugs (a flush lost-wakeup + a WAL delete-vs-upload resurrection race). Reproduce with GOMAXPROCS=2 + -race + -count before dismissing or widening a timeout; escalating the debugger (opus→fable) found the race that timeout-widening had masked.
- 2026-07-04: Treat every debugger/investigator fix as a HYPOTHESIS until CI (the real env) confirms — local-only green is weak evidence for env-specific failures. Two successive fixes passed locally (incl. an injected repro) yet CI disproved both; re-pushing to let CI adjudicate is what exposed the true root cause.
- 2026-07-04: When "fix X" silently grows into an out-of-plan production change (here: a durability bug fix in WAL truncation), STOP at a gate and get explicit sign-off before committing prod code — don't let a test-greening task expand into shipping concurrency fixes unannounced.
- 2026-07-04: Tooling — `gh pr edit` can fail (exit 1) on the Projects-classic GraphQL deprecation; update title/body via `gh api -X PATCH repos/O/R/pulls/N -f title=... -F body=@file`.
- 2026-07-04: When a subagent's final report is garbled/unusable (e.g. it punts to a nested monitor), reconstruct state from git/tests directly rather than re-trusting the prose.
- 2026-07-04: For "add observability/logging/metric/error-handling to component X" tasks, dispatch a convention-scan searcher FIRST — extract the repo's existing pattern (logger-field-vs-slog.Default(), Options struct, log levels for retried-vs-fatal errors) and lock the coder spec to a named peer component. This run, matching the GC-collector / WAL-flusher logger pattern exactly gave the reviewer zero convention nits. Design-from-convention beats design-from-first-principles for hygiene adds.
- 2026-07-04: Observability added to a high-frequency poll loop must use TRANSITION logging, not per-tick: first-failure-of-streak → Warn, continued → Debug, recovery → Info (loop-local bool, not a struct field). Satisfies "persistent failure must be visible" WITHOUT spamming at the poll rate (250ms here = 4 logs/sec of naive Warn). Settle this tension in the spec, before the coder writes it.
- 2026-07-04: Extend the collapse-rule (was: bug fixes + audits) to SCOPED HYGIENE/OBSERVABILITY adds. SnapshotPinner logging ran clean as: scope-gate (1 AskUserQuestion) → convention searcher → TDD coder → revert-discriminate reviewer → gate. No brainstorm / design-doc / plan-doc / worktree ceremony.
- 2026-07-04: While blocked waiting on a background agent, emit a plain text turn — do NOT speculatively load tools (burned 2 turns loading TaskGet/Monitor unnecessarily this run). The harness notifies on completion regardless.
- 2026-07-04: Collapse-rule for DELETE-DEAD-CODE / DOC-ONLY changes: "no new test" is the CORRECT outcome when nothing new is testable. Verification = build + vet + existing-suite green PLUS a reviewer pass whose load-bearing check is comment/claim ACCURACY against the code (does the doc overclaim?) — this replaces the usual revert-discriminate, which is inert when there is no new test to revert. GC ErrUnsafeDelete cleanup ran clean this way.
- 2026-07-04: A subagent's completion report may HTML-escape code in its prose (`&gt;` for `>`). Before flagging it as a defect, verify the actual FILE with Read, not the report text — this run a comment's `>=` showed as `&gt;=` in the report but was correct on disk. Cheap check, avoids a spurious coder round.
- 2026-07-04: After a worktree branch's PR squash-merges, do NOT reuse that branch for the next task — the squash leaves the branch's original commits as stale duplicates, so a fresh PR re-diffs merged content. Start the next task on a fresh branch off origin/master (or commit then cherry-pick the new commit onto one). Operationalizes the 2026-07-03 clean-PR lesson as a START-of-task action, not just a pre-`gh pr create` fixup.
- 2026-07-04: Reconcile-before-picking is NOT enough on a long session — RE-FETCH origin/<default> immediately before commit/PR too. This run B4 was genuinely open at pick-time, but a parallel maintainer session merged the identical fix (#34) 9 min before I committed; I pushed a fully redundant fix PR because I never re-fetched before `gh pr create`. A concurrent merge can overtake a multi-agent session between pick and finalize. Cheap guard: `git fetch origin && git log --oneline HEAD..origin/<default> | grep -i <bug-id>` right before finalizing; if the work already landed, repurpose (e.g. keep only the deterministic regression tests the merged fix lacks) instead of shipping a duplicate. Extends bullets 5/12 (fetch-before-reconcile) to fetch-before-finalize.
- 2026-07-04: A "document + accept the risk" scope pick is INVALID if a test already asserts the invariant the risk would violate. Before offering/choosing document-vs-fix at a scope gate for a known race/edge, grep the test suite for an existing invariant/stress test that exercises it (esp. adversarial configs like GcGraceWindow=-1ns). If one exists, "accept" means shipping a RED test — the fix is already forced. This run, the GC TOCTOU we chose to "document + accept" had a stress test (TestInvariant_AdjacentSortedRunsCompactedAndDeleted) that made accept impossible; a test-scan at scope-time would have pre-empted a full reversal (document PR → debugger → re-check fix, all after CI went red).
- 2026-07-04: When a "pick next task" PR triggers CI, check TRUNK's own latest CI health FIRST. Master was already red (a flaky GC race, newly caught once #34 made -race CI live). A logic-free doc PR inherited that red — not a regression in its diff. Reconcile CI health, not just commit lineage (`gh run list --branch <default>`), so "my CI fails" is framed correctly instead of chasing a phantom regression in a clean diff. Then reproduce the real race (debugger, GOMAXPROCS + -race) before treating it as a flake.
- 2026-07-04: Harness-injected <new-diagnostics> can be STALE — captured mid-subagent-run, not at read-time. A compile error + a phantom _test.go file diagnostic contradicted a debugger's "clean tree" claim; `git status` + `go build` proved the tree clean. Verify against git/build, not the diagnostic snapshot — the "trust git, not agent prose" rule extends to harness diagnostics.
- 2026-07-04: Design-premise contradiction between subagents on an "upstream/reference does X" claim → settle by READING the reference source before rewriting, not by another agent round. This run a first Explore agent said the compaction expiry pre-walk runs PRE-fold; the reviewing architect said POST-fold. One read of upstream `compactor_executor.rs` settled it (post-fold, inside RetentionIterator) and killed v1's entire pre-fold `expiryFilter` premise before I rewrote in the wrong direction. Extends bullets 3/6 (verify prescriptions; one source observation settles opposed agents) to the DESIGN-PREMISE case: when the disputed thing is the design's load-bearing assumption, the source read is far cheaper than the rewrite.
- 2026-07-04: For a "faithful port of upstream behavior X" fix, probe at DESIGN time: "is the state this fix READS actually populated/advanced in our implementation?" B3's compaction cutoff was upstream's persisted `last_l0_clock_tick`, but Go never advanced that tick on flush (default 0 forever) → a byte-faithful cutoff would be permanently INERT (green tests, zero effect). Catching it at design — not after a green-but-does-nothing impl — reframed "compaction-only" into a write-path prerequisite and became a scope-fork gate. Checklist item: for every input a fix consumes from persisted/shared state, confirm some code path in OUR impl writes it with a real value.
- 2026-07-04: Batch coders by DISJOINT package and serialize their commits on the shared branch — no per-coder worktree isolation needed when tasks touch non-overlapping files AND you don't run them concurrently (the hazard is parallel git-index writes, not shared files per se). B3 ran 6 sequential coders across a memtable/flush track + a compactor track clean this way. Refines bullet 8 (which mandates isolation/serialize for SHARED signatures) with the disjoint-file case.
